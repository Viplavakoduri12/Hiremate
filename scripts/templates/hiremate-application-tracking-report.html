4. React Code Used for the Applicant Tracking Timeline

The tracking screen in TrackStatus.jsx defines fixed status steps, fetches the current user's applications, converts backend records into display data, and calculates progress percentages for the timeline. This is one of the most important code sections for the application-tracking feature.

Tracking steps, fetch logic, and progress calculation from the website
const steps = [
  { key: "APPLIED", label: "Applied" },
  { key: "UNDER_REVIEW", label: "Review" },
  { key: "SHORTLISTED", label: "Shortlisted" },
  { key: "ACCEPTED", label: "Selected" }
];

useEffect(() => {
  fetch(apiUrl("/api/applications/my-applications"), {
    headers: {
      Authorization: `Bearer ${localStorage.getItem("token")}`
    }
  })
    .then(res => res.json())
    .then(data => {
      const formatted = data.map(app => ({
        jobRole: app.role,
        status: app.status,
        createdAt: app.createdAt,
        rejectionReason: app.rejectionReason || ""
      }));

      setApplications(formatted);
    })
    .catch(err =>
      console.error("Dashboard fetch error:", err)
    );
}, []);

const getProgressPercent = (status) => {
  if (status === "REJECTED") return 25;

  const index = steps.findIndex(
    (s) => s.key === status
  );

  if (index === -1) return 0;

  return Math.round(
    ((index + 1) / steps.length) * 100
  );
};
  • The timeline logic converts raw backend data into a clear user-facing progress display.
  • The progress bar is tied directly to the ordered step list.
  • The same screen also calculates summary totals for review, shortlist, and accepted candidates.

6. What Assertions and Validation Points Suit This Website?

Assertions for HireMate should verify both the visible UI and the business state behind it. The following points fit the current website structure and can be used for manual testing, UI testing, or Selenium-style automation.

  • Homepage Rendering Assertion: the landing page should show the HireMate headline, workflow cards, and the three case-study PDF cards.
  • PDF Link Assertion: each case-study card should point to its own PDF file in /case-studies/.
  • Protected Route Assertion: applicant, HR, and admin pages should not open without a token and matching role.
  • Tracking Progress Assertion: status values such as APPLIED or SHORTLISTED should map to the expected progress percentage.
  • Summary Count Assertion: the Track Status page should compute total, review, shortlisted, and accepted values from application data.
  • PDF Export Assertion: the HR shortlist report should generate a PDF with header, footer, summary cards, and candidate rows.
  • Error Message Assertion: fetch failures or invalid operations should surface a readable message rather than silently failing.

7. Execution Notes and Identified Improvements for HireMate

A. Observed Website Strengths

  • The homepage clearly communicates the product to both job seekers and HR recruiters.
  • The case-study area is already integrated into the public website.
  • The tracking page uses a clean step-based model that is easy for applicants to understand.
  • The HR dashboard contains a thoughtful PDF export feature with branded layout and pagination.

B. Improvements Identified During Code Review

  • Protected Route Trust: route access currently depends on localStorage token and role values, which is fine for UI control but should always be backed by secure server-side authorization.
  • Tracking Fetch Feedback: the track screen logs fetch failures to the console, but a stronger user-facing empty or error state would improve resilience.
  • Encoding Cleanup: a few files contain odd encoded symbols in comments or text output, which should be normalized for cleaner maintenance.
  • Screenshot Automation: report assets are easier to maintain if website screenshots are regenerated whenever the UI changes.
